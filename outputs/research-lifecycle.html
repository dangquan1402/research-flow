03 Experiment Lifecycle

This is the core iteration loop. Every experiment starts with a question, produces a result, and feeds back into the knowledge base. The loop runs until the question is resolved or the research direction changes.

04 Stage Details

1. Init — /research

Define the research goal. Create a GitHub issue. Create the branch (research/GH-{id}-{slug}). Scaffold memory structure.

2. Ingest — /analyze

Feed a source (paper, URL, dataset). Extract entities and findings into memory. Log the operation.

3. Analyze — /analyze

Read the goal + accumulated memory. Identify gaps. Generate new analysis. Surface open questions that need experiments.

4. Experiment — /experiment

A Pick Question

Select an open question from memory/open-questions/. Define a testable hypothesis with acceptance criteria.

B Create Config

Write a YAML config in experiments/configs/ with meta.question, meta.hypothesis, and meta.acceptance_criteria.

C Run Experiment

uv run python -m experiments.run config.yaml — trains the model, logs to MLflow, writes results JSON and run-log entry.

D Classify & Record

Classify outcome (success/partial/inconclusive/failed). Write finding to memory. Update log and index. Plan next iteration if needed.

5. Synthesize — /synthesize

Consolidate findings into themes. Resolve contradictions. Cross-reference related findings. Produce synthesis output.

6. Distill — /distill

Extract actionable decisions from settled findings. Create decision records in memory/decisions/. Update or create baseline configs in experiments/configs/baselines/.

7-10. Evidence, Verify, Critique, Lint

Generate charts backing claims. Write code to verify intermediate results. Adversarial review for gap analysis. Health-check memory for orphans and contradictions.

05 Failure Taxonomy

Every experiment gets classified. Negative results are first-class findings — they narrow the search space and prevent repeating dead ends.
StatusMeaningActionCreates Finding?
success Met acceptance criteria Write positive finding, may close question Yes (positive)
partial Completed, missed criteria Write negative finding, plan next iteration Yes (negative)
inconclusive Results ambiguous Record, may re-run with more epochs or different seed Yes (inconclusive)
failed:config Bad config (OOM, shape mismatch) Fix config, re-run No
failed:bug Code bug crashed training Fix bug, re-run No
failed:infra Infrastructure issue (disk, memory) Re-run when stable No

06 Tooling & Infrastructure

uv

Package management. uv sync to install, uv add to add deps, uv run to execute.

MLflow

Experiment tracking. Params, metrics, model artifacts. --mlflow flag on trainer.

MLX

Apple Silicon ML framework. Native Metal acceleration. All training runs on M4.

YAML Configs

Experiment definitions in experiments/configs/. Baselines, sweeps, with metadata linking to questions.

Run Log

Append-only JSONL at experiments/results/run-log.jsonl. Every run tracked with status and metrics.

Git Flow

research/ → hypothesis/ → synthesis/ branches. Each tied to GitHub issues.

07 Memory Structure

memory/
● index.md Catalog of all pages (auto-updated)
● log.md Append-only chronological record
▶ entities/ Papers, researchers, concepts, tools
▶ findings/ Experiment results + analysis insights
● experiment-*.md Each experiment gets a finding (positive, negative, or inconclusive)
▶ themes/ Cross-cutting patterns across findings
▶ open-questions/ Gaps identified, queued for experiments
▶ decisions/ Distilled actionable decisions
● decision-*.md Links evidence findings + baseline config
experiments/
▶ configs/ YAML experiment definitions
▶ baselines/ Verified best configs per task
▶ sweeps/ Multi-variant sweep definitions
▶ results/ Experiment outputs
● *.json Per-run results with epoch logs
● run-log.jsonl Append-only registry of all runs
▶ training/ MLX trainer, evaluator

Finding Types

Positive Finding

Experiment met criteria. Records: Insight, Evidence, Key Observations, Counter-arguments, Implications.

Negative Finding

Experiment failed criteria. Additionally records: Why It Failed, What This Rules Out, What This Suggests Instead.

Decision Record

Distilled Decision

Created by /distill from multiple settled findings. Contains: Decision statement, Rationale, Tradeoffs Accepted, Revert Conditions. Links to evidence findings and a baseline config YAML.

08 Example: A Full Research Cycle

Tracing one question through the complete lifecycle, using actual results from this project's arithmetic transformer research.

Open Question: "Can a 2-layer model with scratchpad solve 3-digit multiplication?"

File: memory/open-questions/optimal-multiplication-scratchpad-format.md

Hypothesis: "Scratchpad enables 2L/384D to achieve >99% on 3-digit multiplication"

Config: experiments/configs/baselines/mul-3d-scratchpad-2L384D.yaml

meta:
  name: "mul-3d-scratchpad-2L384D"
  question: "optimal-multiplication-scratchpad-format"
  acceptance_criteria: ">99% accuracy on 3-digit multiplication"

task:
  op: mul
  max_digits: 3
  scratchpad: true

architecture:
  n_layers: 2
  dim: 384
  ff_dim: 1536

Command:

uv run python -m experiments.run experiments/configs/baselines/mul-3d-scratchpad-2L384D.yaml

Result: success — 100% accuracy, 3,567,744 params, ~3000s training

Run log entry appended to experiments/results/run-log.jsonl

Finding: memory/findings/experiment-scratchpad-2L384D.md

"Scratchpad pushes 2L/384D mult from 85% to 100%. Scratchpad eliminates the depth bottleneck — 2L/384D is now a universal architecture for all single-operation tasks."

Decision (after distillation): "Use 2L/4H/384D + scratchpad as the standard multiplication architecture. SwiGLU optional for faster convergence."

Baseline config verified and tagged ["baseline", "verified"].

Open question status updated to resolved. The scratchpad format question spawned further research into 5-digit multiplication (77.85% — partial result, iteration continues).
